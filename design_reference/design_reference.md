## 一、产品背景与核心需求

前段时间我跟小伙伴们开会，我发现对于我们这么小的一个团队，对于所谓的工作安排这件事来说，居然每个人的说法都不一样，没有人能够绝对地说清楚我到底什么时候该出差、什么时候该回来、什么时候该录片子、什么时候该出去参加一些行业活动。

我就问说为什么我们以前没有这样的困扰，小伙伴们告诉我说因为以前有秘书，现在不具备请秘书的条件了。于是我们的话题就开始转向：到底有没有一个这样的工具，可以彻底一劳永逸地解决日历协作的问题。

通过半个月时间的寻找，我试了市面上各种各样的日程管理软件，我发现有一个共同的问题——每一个软件，当你想要去创建一个新的日程的时候，都必须要点那个加号，填写一个标题，选择时间，选择地点，选择月份，选择参会人等等。等你填完那个表格的时候，你已经忘了你在干嘛了。更痛苦的是每一次你要想修改一下，刚才所有的流程要重走一遍。

于是乎，我突然有个想法：在今天大模型推理能力这么强、AI 编程也已经非常强大的情况下，有没有一种可能，我们自己做一个这样的日历协作软件？

## 二、产品核心理念：LUI（基于大模型的用户界面）优先

我脑子里想象的画面就像一个微信的对话框一样，你可以不停地往里边丢需求，然后充分相信后边有一个聪明的脑袋帮你把所有的问题都捋顺，给你一个漂亮的结构化的结果。

**典型交互示例：**

- "4 月 18 号，我要去参加一个 AI 行业的研讨会，你帮我记录一下"
- "你帮我看一下今年北京车展是什么时候，我要去参加新闻媒体日，提前两天到北京，媒体日结束之后回深圳"
- "下礼拜一我不想上班了，不去公司了，把所有的行程都给我清空，然后帮我请假"
- "明天看一下我什么时间有空，我要跟团队开一个新年的业务规划的会议，帮我找个时间安排一下；如果实在没时间的话，就清掉一个会议，下班以后安排"
- "这是我前几天去上海出差的代驾的发票，是在新荣记吃完饭之后叫的代驾，你帮我关联一下，然后帮我报销"
- "我今天给团队买咖啡，花了 150，帮我记一下"

刚才所有的 demo 全部都是真实的 API 请求，平时我在自己真正用的时候是用真机，这是真正的生产环境。刚才这套是同一个代码但在本地环境，所有的 API 请求都是真实的。

这就是过去 40 天我做出来的第一个 **AI Agent**，解决三个问题：
1. 日程管理
2. 费用管理
3. 把它们两个有机地结合在一起

目前产品状态：在 **TestFlight** 上，身边的朋友们在小范围试用。

## 三、为什么现有工具不够用

### 3.1 为什么不能直接用大模型聊天界面

大模型有**上下文窗口（context window）**的局限。你一旦创建的日程超出了这个窗口的范围，记录立马就被截断了。而且当你的日程在高速高频地调整时，大模型很难在几周以后依旧帮你精准地定位并做精准的修改。所以大模型并不能解决这个问题，因为大模型没有结构化的数据。

### 3.2 为什么 Siri 不行

Siri 唯一能处理的叫做单一 CRUD（增删改查）——你要么创建一个会议，要么删除一个会议，要么把某个会议改到某个时间里面去。你要想真正完整地进行多意图批量操作、结合互联网搜索再去做，Siri 是绝对做不到的。再说他的日历里边根本不具备所谓的费用管理追踪，甚至是整合和生成的能力。

### 3.3 为什么其他工具不行

所有的其他工具基本上都是 **Web 2.0** 时代的产品，你会发现几乎每一个记账软件，当你点开那个加号的时候，都要先选择娱乐、餐饮费用、日常生活费用等等，然后再输入一个金额，然后再点确认。和日历创建一样，都要经过一个漫长的填表的过程。

为什么所有的这些产品在他们创建之初是这样？因为没有大模型，只能通过一个个表单、图标、图形界面的方式，来将你脑子里那些模糊不清的表达结构化，因为只有结构化的数据才能够被良好地存储在数据库当中才能被查阅管理。

这些产品也有增加一些 Agent 的能力，其实有，无非就是在上面绑了一个聊天机器人（Chatbot），但是这些 Chatbot 跟他们的数据库之间其实是高度隔离且不能无缝操作的。也就是说它不是一个原生的 AI 产品，而是一个旧时代的产品封上了一层 AI 的套壳。

### 3.4 为什么 Rewind（类"龙虾"全记录工具）不行

它之所以强大，是妥协了两个重大领域所换来的——
1. **隐私性**：你需要在你的机子上面开放几乎所有的权限它才能强大；
2. **经济性**：无限的本地记忆存储，即便你做大量的记忆读取优化，也会经常烧干你的算力。

解除了隐私和成本的封印之后让你看到今天大模型可能的上限在哪里，但一个以妥协隐私和不考虑经济性为前提的工具，是不具备真正上架和投放市场的条件的。

## 四、产品架构三部分

在我的脑海当中，这个产品其实等于三个部分：
1. **大模型（LLM）**：脑子，负责思考
2. **Agent**：负责在本地数据库或各种各样的工具当中来回调用，不停推理理解用户需求，完成固定的任务
3. **图形界面（GUI）**：即便可以在聊天对话当中解决几乎所有的 CRUD、批量操作、费用记录，也依旧难以解决大模型的延迟问题——有些时候我们只是很简单地想看一看明天有什么，快速查询的图形界面依旧非常重要

**设计原则**：图形界面不负责输入，只负责数据呈现。

## 五、图形界面设计

### 5.1 列表视图（主界面）

- 滚动菜单展示所有日程
- 快速上下滚动，显示距今还有几周
- 下方有箭头可快速回到今天

### 5.2 日历视图

- 支持 Day View（日视图）
- 支持 Monthly View（月视图）
- 点击任意日期直接在下方看到当天日程
- 右侧有**滚轮（Dial）**，有惯性，可滚动切换日期

### 5.3 费用视图

- 与日历视图使用相同逻辑，"把钱放在时间里面管理"
- 当前视图下方不是事件，而是本月所有报销费用的汇总
- 支持点开单笔费用查看关联日程（如：星巴克 16 块 → 关联北京出差日程）
- 生成报销单是简单的后续步骤（一旦钱与事件匹配完毕）

**核心理念**：将 LUI（Language User Interface，基于大模型的用户界面）放在前面作为整个交互的核心，同时通过图形界面给用户一个快速查询的入口。

## 六、工程架构原则

### 6.1 薄客户端（Thin Client）原则

所有的 CRUD、Agent 循环、工具调用，**没有一个发生在 iOS 端**。iOS 只是一个单纯的显示界面，每个卡片的渲染，iOS 前端就像一个传菜员一样，只负责将后端的指令进行前端渲染。

甚至一张卡片里边显示什么字、该渲染什么字段，都是由后端的指令直接发给 iOS，**iOS 做"无脑渲染"（Dumb Rendering）**——厨子告诉我说这叫牛排，iOS 只负责把它端到用户面前，甚至连牛排是什么东西做的都不知道。

让 iOS 和业务逻辑之间完全剥离。

**为什么选择薄客户端：**
- 当你想拓展平台（Android、桌面端）时，只需要写前端交互界面就可以了，所有的业务逻辑不会因为平台拓充而受到影响
- 当你想增加业务逻辑和能力时，只需要在云端部署，所有的平台会同步进展
- 无论是从运维的角度还是从未来平台适配的角度，都是更高效率的一种方式

### 6.2 iOS 端技术选型：原生（Native）+ Web 混合

**只有聊天界面是原生 iOS（Native iOS）：**
- 滑动组件
- 卡片的选择
- 输入框位置
- 按住说话（语音输入）
- 全屏遮罩
- Photo Picker（图片选择抽屉）

**日历视图等是 Web 页面（WebView）：**
- 依旧能有 Haptic Engine（震动引擎）反馈
- 通过桥接的方式和 iOS 原生硬件能力结合在一起
- 可以达到几乎原生的状态，普通消费者感受不出是 Web 页面的差别

**为什么日历视图选 Web 而不是原生 iOS：**

> AI 写代码时原生 iOS 的代码质量实在太低，不管是用 **MVVM** 还是 **Store Reducer 架构（如 TCA）**，都会有玄学 bug 出现。这是因为 iOS 本身生态的封闭性，大模型依赖海量数据来做训练，iOS 的相关训练数据相对较少，生成质量差。
>
> 如果完全依赖 AI 的水平来做原生 iOS，光一个日历界面就能做两个月；但如果用 Web 语言，即便是复刻如此复杂的关联交互，只写了一下午，又花了一天时间做交互微调。这就是 Web 和 iOS 之间代码生成质量上的巨大差异。

## 七、DDD（领域驱动设计，Domain-Driven Design）

### 7.1 两个完全独立的业务域

- **日程管理域**：时间、地点、人物、事件管理
- **费用管理域**：完全另外一套规则

它俩甚至都不知道彼此存在。

### 7.2 编排器 Agent（Orchestrator Agent）

跨业务域的协调和查询由**顶层编排器 Agent** 来做：
1. 编排器负责同时下发任务给两个 Agent
2. 两个 Agent 自己处理完毕，写完数据库（DB），把结果返回
3. 顶层 Agent 再呈现出结果给用户

### 7.3 DDD 的优势

- **独立部署**：如果有人只喜欢日历管理，不需要费用追踪，可以直接给他单独这一个模块
- **业务语义隔离**：每个域按照自己的业务板块需要任意发挥，不用担心两者之间相互污染
- **高内聚、低耦合（甚至完全不耦合）**

### 7.4 当前重构计划

目前代码当中有一丁点儿耦合程度，正在做完全的重构——**在上面再加一层抽象层（Abstract Layer）**，当你再加第三个、第四个业务域的时候，可以完全做到通过配置甚至少量的代码就可以增加一个新的业务域，而不影响现有 Agent 的表现状态。同时表结构也是完全隔离的。

> **架构核心**：DDD 领域驱动 + 共享基础设施（Shared Infrastructure），是整个 app 未来可拓展性的核心。

## 八、Agent 架构演进（三个阶段）

### 阶段一：大模型作为意图解析器（错误方向）

最开始只把大模型当做一个**意图解析器（Intent Parser）**：
- 用户说"明天帮我三点钟创建个会议"
- 大模型解析意图 → 手写大量代码和功能来执行
- 实际上并不是严格意义上的 Agent，而只是把大模型当解析器，靠代码脚本来完成用户意图

**问题**：丧失了 Agent 最重要的价值——**灵活性**。脚本必须从一跑到十跑完，中间打断一次、停留一次完全不行。

### 阶段二：ReAct（Reasoning + Acting，推理-执行循环）模式（另一个极端）

完全使用 **ReAct** 逻辑：**推理 → 执行 → 推理 → 执行 → 无限循环**，每次只做一件事。

**问题**：灵活性够了，但效率低。如果想批量删除四个事件，就要第一个事件推理完成、第二个事件推理完成……做几个事件时间就要乘几倍。

### 阶段三：计划与执行流水线（Plan & Execute Pipeline）——最终选择 ✅

不管用户想要批量创建还是单一创建，在我看来都是**一个计划（Plan）**，只不过这个计划包含三个动作还是包含一个动作。

**流程：**
1. **大模型统一推理**：统一解析时间、统一解析地点，生成完整计划
2. **呈现给用户确认**：用户点击确认这个计划
3. **执行阶段**：开始操作数据库

**核心洞察**：
- 消耗算力最多的是**推理和制定计划**的过程
- 计划一旦确定，写入数据库其实是毫秒级的事情
- 换取了巨大的用户体验提升

**对比**：原来批量创建五个事件大概要一分半到两分钟，模型不停推理推理推理，还会涉及上下文溢出、不连贯、甚至会漏掉等情况。

## 九、Smart Agent + Dumb Tools 原则

充分相信大模型的推理和计划能力，而把**工具（Tools）做得非常极致和简单**。

**错误做法（重构前）**：
- 工具里边加上校验逻辑、解析逻辑、写入数据库的逻辑
- 工具特别"聪明"但功能不够单一、不够纯粹
- 所有的判断发生在工具内部，是个黑盒，不了解它，也不知道 bug 在哪里

**重构后**：
- 直接删掉了将近八九千行代码
- 把创建工具、删除工具、更改工具全部变成非常"蠢"的**执行器（Executor）**
- 把所有的校验功能全部剥离，交给 Agent 和大模型掌握足够多的上下文信息，去做充分的推理判断
- 工具只做最愚蠢、最简单的执行

## 十、可观测性（Observability）与工程质量

### 10.1 为什么需要可观测性

整个项目 12 万行代码里边，有将近 30% 是跟功能无关的，主要解决**可观测性和工程质量问题**。

**问题场景**：对于一个非计算机专业背景的人来说，如果每次测试 bug 都让 AI（如 Claude Code）扫所有代码，会导致：
1. 浪费极多的 Token，迅速 Token 爆炸，上下文一爆掉大模型会压缩上下文，越压缩越不准——无限恶性循环
2. 扫的过程非常慢，可能要扫八到十分钟，一上午只能改三个 bug
3. 对于跨领域的多 Agent 架构，这样的时间开发成本是不可承受的

**解决方案类比**：如果一个 Agent 就像击鼓传花，从第一个传到第十个，第三个把花掉到地上了，我们要迅速锁定是第三个。可观测性就是把这十个人每个人编上号，把他们的行为模式也编上号。当出现 bug 时，直接让大模型根据编号迅速找到相关的节点。

### 10.2 决策 ID 追踪系统

- 在代码量 1 万行时就开始建立 **Logging（日志）系统**
- 目前大概有 3~4 万行代码量
- 大模型在搜的过程中直接锁定关键 log、直接锁定关键结果
- 帮助大模型迅速解决问题，而不是人工看 log

### 10.3 自改进飞轮（Self-Improvement Flywheel）

1. 生产环境用户遇到 bug → 查 log
2. 自动把这些 bug 写成**测试用例（Test Cases）**
3. 在本地环境下复现，忠实反映当时用户的真实情况
4. 整个过程做成两个自动化工作流，加入上线测试过程

### 10.4 回归测试（Regression Testing）

- 写了 100 多个测试用例（Test Cases）
- 通过 **WebSocket** 的方式直接投送到 iOS 模拟器（Simulator）里
- 可以看到每个 case 在模拟器里面跑
- 跑完了之后在 terminal 或 Claude 里告诉你过还是不过
- 不仅能看见数据结构是否完整、数据校验是否正确，最主要是能看到 **UI 渲染是否正确**

**上线流程**：每次上线之前花一个多小时，把 98 个 case 跑完，看每个卡片都对，然后再进行上线，确保没有回退（Regression）。

在做这个**测试 Harness（Test Harness）**的时候，参考了大量行业最佳实践（Best Practices），比如 Anthropic 和 OpenAI 是怎么做 Agent 测试的。

## 十一、是否过度设计（Over-Engineering）？

这套产品是一个 **Multi-Agent System（多 Agent 系统）**。是否在架构上过度设计，取决于几个非常重要的因素：

1. **你的项目愿景**：你到底想做的是一个自己用的小工具，还是希望真正上到生产、投放市场的大规模应用产品？
2. **未来对于这个产品的规模和上限设计在哪里？**
   - 如果只是自己用，让它 work 就可以，完全可以做得非常简单
   - 如果要考虑生产环境的拓展或功能的拓展，必须要将架构设计得足够有前瞻性
3. **注意反向情况**：你在做一个 Agent 的时候可能会发现架构是过度设计，但很快就会发现它其实是设计不足

> **结论**：是否过度设计，这个问题是需要在产品端、策略端想清楚的，而不是一个技术问题。

## 十二、设计体系（Design System）

### 12.1 Tokenized Design System（令牌化设计体系）

整个系统采用**令牌化（Tokenized）**的设计体系，大概有 1 万多行设计体系代码。

把以下元素全部变成一套 **Design Token（设计令牌）**：
- 字体、字号
- 颜色
- Spacing（间距）
- 阴影材质（玻璃、磨砂玻璃、半透明等）

通过 **JSON Token Loader** 加载到每一个组件上去。

**好处**：只需要做一次——选定颜色、色号、间距，一旦成为规范，做任何一个组件、任何一个界面，只需要让它读取现在的规范就可以迅速生成，而不需要每个卡片重新设计一次。

### 12.2 工具链

最终沉淀下来的设计工具：
- **Penpot（MCP 集成设计工具）**：如果有新想法、想换颜色、想做品牌视觉识别体系（Brand Identity），可以直接在 **Claude Code** 里输入"打开 Penpot，帮我做一套登录界面"，它就会直接在 Penpot 里边画出来。解决了 CLI 界面当中不能进行实时预览的问题。
- **Claude**：AI 编程助手
- **Mobbin**：寻找设计灵感

尝试过的其他工具：**Figma**、**Shadcn**，最后选择了 Penpot 作为主力设计工具。

### 12.3 UI 参考

日历界面参考了 **Time Page**（App Store 中的一款极高品质日历设计产品），在此基础上做了自己的调节。但如果产品要上线收费，整个 Timeline 界面需要全部换成原创设计。

> Time Page 的问题：1. 是外国人设计的，跟中国人的生活习惯不搭调；2. 依旧逃避不了需要填表——只是把表格打印得漂亮了一点。

## 十三、技术选型

### 13.1 Agent 框架：Vercel AI SDK

- 花时间最多、代码量最集中的部分
- **Vercel AI SDK**：开箱即用、功能极为强大的 Agent SDK
  - 帮你处理工具循环（Tool Loop）
  - 基本的上下文管理（Context Management）
  - 结构化输出（Structured Output）
  - 等做 Agent 的必备功能
- 对于刚刚入门做 Agent 开发的人来说，可以解决大量摸索和探索的时间
- **Google ADK（Agent Development Kit）**：谷歌出的 Agent 开发套件，对谷歌自己生态的支持性更好，但对于不想在一个生态当中绑死的人，选择 Vercel AI SDK

### 13.2 数据库

数据库选型略显 **overkill（超配）**，但是为了：
- 未来规模化的拓展
- 多租户（Multi-Tenant）的拓展
- 绝对安全稳定的数据

产品本质就是：**大模型 + Agent + 数据库**，如果不在 Agent 和数据库这里做足够的预留，未来的技术债是非常痛苦的。

### 13.3 基础设施

- **Docker Compose**：没有用 **Kubernetes（K8s）**，为了大幅度加快研发节奏
- **iOS 端**：部分原生 iOS + 部分 Web 混合（理由见第六节）

### 13.4 API 选择

- **Anthropic 全家桶**：classifier、orchestrator（编排器）、schedule agent、expense agent，所有循环全部使用 **Claude Sonnet 模型**（根据需求使用不同版本）
- **高德地图 API**：
  - Input Tip（地址补全/自动补全）
  - 地址精准搜索
- **千问（Qwen）Embedding 模型**：每一个日程创建完毕后，在后台用千问的 embedding 模型把它**向量化（Vectorization）**，然后做向量匹配，搭建了一套**混合打分（Hybrid Scoring）**系统
- **Jina Reranker**：当所有评分比较接近时，用 Jina Reranker 做最终的评判

## 十四、开发复盘与反思

### 14.1 开发数据

- 项目持续：约 **48 天**
- 总共写了约 **20 万行代码**，最终有效代码约 **12 万行**
- 不停地重构是时间消耗的主要原因
- 开发利用业余时间（出差路上、晚上出席活动回来后、录完片子后）

### 14.2 浪费的时间分析

1. 非全力投入（not all-in）导致效率损失约 40%
2. 对很多架构、原则的理解是在一边做的过程中一边磨合的，浪费了大量时间和精力

> 如果有一个有经验、有架构思维的程序员，在理解了产品愿景和目标的情况下，可以比现在这套做法快 **5~8 倍**（估计专业开发者可以达到 7~8 倍效率提升）。

### 14.3 核心感悟

> Claude Code 的创始人讲到过一句话：**"Coding is largely solved today."**（今天写代码这件事在绝大部分程度上已经被彻底解决了。）
>
> 甚至未来"软件工程师"这样的一个名称都已经不再存在，会有更多的 **Builder（构建者）** 涌现出来——他们可能过去并不具备写代码的经验和能力，但因为他们对于产品的愿景、对自己的学习能力、对 AI 工具驾驭的纯熟度，以及超强的消化和理解力，会迅速弥合传统软件开发者和普通人之间的距离。
>
> 创造力会重新占领高地，对用户体验的理解会重新占领高地，对市场敏锐的洞察会重新占领高地。
>
> 在今天高度发达的互联网环境当中，没有哪一个需求是没有工具去解决它的，只不过我们要思考的问题不是去填补空白，而是如何用新时代、新工具、新思维去提出一个更加优秀的解决方案。
